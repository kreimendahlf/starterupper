Introduction: Learn git and project hosting jargon

Git is a version control system: it stores snapshots of a folder over time into a repository. To record a snapshot, you must decide what changes in the working directory to add to the staging area. Snapshots, known as commits, record the timestamped contents of the staging area, the author's name and email, a descriptive message, and references to ancestor commits. Making a commit happens locally; it does not submit work over the network. The SHA1 hash of the commit uniquely identifies it in the repository, and verifies integrity.

Git allows history to split off along separate timelines called branches, which can merge back together. The default branch is called master. To see the working directory as it was at another point in time, checkout a commit or a branch; HEAD points at what is checked out. During a merge, a merge conflict may happen when two branches have different changes to the same regions within a file; use a merge tool to fix these.

Git is distributed in that every repository has a list of remote repositories, each with a nickname and URL. The default remote is called origin. Project hosts such as Bitbucket, Github and Gitlab offer storage for remote repositories and manage access to them over the network. To submit work, push a branch to a remote repository. To receive updates from a remote repository, fetch then merge, or equivalently, pull.

Hints for using git
  • Commit early and often: keep commits small
  • Use branches for experiments
  • Test before you push, don't push if it's broken
  • Push to origin, not to upstream
  • See CONFLICTs? Use a merge tool
Submit work from the Terminal or Git Bash 
cd ~/REPOSITORY
git gui &
git add .
git commit -m "Message goes here"
git push origin master
Receive updates following these steps 
cd ~/REPOSITORY
git commit -am "Message goes here"
git fetch --all
git merge upstream/master
git mergetool
